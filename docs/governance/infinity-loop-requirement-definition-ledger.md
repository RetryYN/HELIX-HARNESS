# HELIX Infinity Loop 要件定義設計台帳

## §0 目的と判定境界

本台帳は要件IDの一覧ではなく、要件定義そのものの設計状態を115件個別に管理する正本である。各行はstable requirement ID、immutable revision、canonical statementの位置とdigest、source atom authority、falsifiable assertion、service/component候補、適用template profile、freeze状態を持つ。

現時点ではsource capability atomizationが未閉鎖であるため、全行のauthorityは意図的に`pending:source-atom-link`、状態は`draft-authority-pending`とする。L1本文やcoverage行が存在するだけで`active/frozen`へ昇格しない。source atom、authority root、acceptance oracle、capability/service、template applicability、design obligationのtyped edgeと独立review receiptが揃った行だけを将来`active`へ遷移できる。

## §1 revision・変更契約

- stable IDの表示名変更だけではrevisionを増やさないが、意味変更は新revisionとdownstream stale receiptを必須とする。
- splitは親の全source atomを子へdispositionし、mergeは全入力atomとacceptance oracleの包含証拠を要求する。
- rename、supersede、reject、N/Aはbefore/after semantic digest、authority、全consumer、rollback/reentryを持つ。
- `challenged`、`deferred`、`authority-pending`、`stale`をfreeze可能として数えない。
- statement digestはL1の要求本文cell UTF-8 bytesへSHA-256を適用した観測値で、L1更新時に再計算する。
