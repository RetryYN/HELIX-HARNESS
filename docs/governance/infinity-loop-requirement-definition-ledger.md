# HELIX Infinity Loop 要件定義設計台帳

## §0 目的と判定境界

本台帳は要件IDの一覧ではなく、要件定義そのものの設計状態を162件個別に管理する正本である。各行はstable requirement ID、immutable revision、canonical statementの位置とdigest、source atom authority、falsifiable assertion、service/component候補、適用template profile、freeze状態を持つ。

現時点ではsource capability atomizationが未閉鎖であるため、125行は`pending:*`、Universal由来16行は
provisional aggregate groupへ結線済みだが`hardening-pending`とする。どちらも`active/frozen`へ昇格しない。
source atom、authority root、acceptance oracle、capability/service、template applicability、design obligationのtyped edgeと独立review receiptが揃った行だけを将来`active`へ遷移できる。

Hybrid Pythonに関する`HIL-FR-15`、`HIL-FR-25`、`HIL-FR-26`、`HIL-FR-27`、`HIL-TR-02`、`HIL-TR-03`、
`HIL-TR-07`から`HIL-TR-10`は、tool SHAをjoin keyとして、Python採用台帳のruntime reuse disposition、source採否台帳の
semantic adoption disposition、L3 `HR-FR-HIL-10/12`、各HIAへ到達しなければactive化しない。現時点は29/29 inventoryだけが
存在し、tool別fixture/contract/Node gate receiptが無いため全edgeをpendingとする。

## §1 revision・変更契約

- stable IDの表示名変更だけではrevisionを増やさないが、意味変更は新revisionとdownstream stale receiptを必須とする。
- splitは親の全source atomを子へdispositionし、mergeは全入力atomとacceptance oracleの包含証拠を要求する。
- rename、supersede、reject、N/Aはbefore/after semantic digest、authority、全consumer、rollback/reentryを持つ。
- `challenged`、`deferred`、`authority-pending`、`stale`をfreeze可能として数えない。
- statement digestはL1の要求本文cell UTF-8 bytesへSHA-256を適用した観測値で、L1更新時に再計算する。
