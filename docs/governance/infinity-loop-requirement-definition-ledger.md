# HELIX Infinity Loop 要件定義設計台帳

## §0 目的と判定境界

本台帳は要件IDの一覧ではなく、要件定義そのものの設計状態を153件個別に管理する正本である。各行はstable requirement ID、immutable revision、canonical statementの位置とdigest、source atom authority、falsifiable assertion、service/component候補、適用template profile、freeze状態を持つ。

authority rootは`infinity-loop-requirement-authority-binding.md`の24 setへ結線済みである。PO chat/directive、accepted ADR、active harness memory、custody済みresearch/source atomを区別して扱い、外部code由来でない要求へsource capability atomを強制しない。L2本文やcoverage行、authority setが存在するだけで`active/frozen`へ昇格しない。acceptance oracle、capability/service、template applicability、primary L3 owner bindingのtyped edgeと独立review receiptが揃った行だけを`active`へ遷移できる。owner bindingはdownstream design obligationのdischargeやL4到達を意味しない。

## §1 revision・変更契約

- stable IDの表示名変更だけではrevisionを増やさないが、意味変更は新revisionとdownstream stale receiptを必須とする。
- splitは親の全source atomを子へdispositionし、mergeは全入力atomとacceptance oracleの包含証拠を要求する。
- rename、supersede、reject、N/Aはbefore/after semantic digest、authority、全consumer、rollback/reentryを持つ。
- `challenged`、`deferred`、`authority-pending`、`stale`をfreeze可能として数えない。
- statement digestはL1の要求本文cell UTF-8 bytesへSHA-256を適用した観測値で、L1更新時に再計算する。
