# コード最小主義（code-minimalism）— 書かない選択肢を先に使い切る

コードを書き始める前、機能追加・依存追加・抽象化を提案する前、ハードコードしそうなとき、
実装量を見積もるとき、生成 AI で大量のコードが安く書けてしまうときに読む。
judgment-core §1-7（スコープ規律）の実装局面での運用形であり、`refactoring` /
`incremental-implementation` と相補する。

## §0 出発点となる態度

**コードは資産ではなく負債である。資産は機能（振る舞い）であり、コードはその維持費だ。**
1 行書くたびに、保守・テスト・レビュー・脆弱性対象面・次に読む人の学習コストが増える。
最強のコードは書かれなかったコードである。

この態度は怠惰の正当化ではない。**書かないためには書くより頭を使う**。エージェントにとって
特に重要なのは、**コードを書くことが最も簡単な逃げ道**だという自覚 — 生成コストがほぼゼロの
AI は、放っておくと何でもコードで解決する。安く書けることは書いてよい理由にならない。
保守・レビュー・攻撃対象面のコストは 1 行も安くなっていない。

## §1 書く前の 7 段の問い

コードを書き始める前に上から順に自問し、**どこで止まったかを作業記録（PLAN / 実装提案の冒頭）に
1 行残す**（「1〜6 を検討の上、7 で実装」の一行が説明責任になる）:

1. **消せないか** — この要求は本当に必要か。「あったら便利」は「なくても困らない」の婉曲表現。
   要件を落とす提案は実装より価値が高いことがある（理由を PLAN に記録して落とす）。
2. **運用で吸収できないか** — 自動化の損益分岐 = 頻度 × 1 回の工数 × 寿命。四半期に 1 回・
   5 分の作業に自動化を書くのは保守費を考えれば赤字。runbook 1 ページで済むなら勝ち。
3. **既にあるのではないか** — 既存 module・framework・標準ライブラリを 10 分調査したか。
   sort・retry・cache・validation・paging はほぼ確実に既にある。inventory-first
   （judgment-core §1-6）はコード 1 行の単位でも適用される。**車輪の再発明は「知らなかった」
   ではなく「調べなかった」**。
4. **設定・データにできないか** — 分岐条件が業務ルールなら、それはコードでなく設定・データの
   仕事。判定基準: 「その値 / 分岐は、リリースなしで変えたい日が来るか」。来るならコードに
   書いた時点で負け。
5. **共通部品にないか** — 似た実装が既に 2 つあるなら、3 つ目を書く前に共通化を検討する
   （behaviour-invariant なら `refactoring` の regression fence の下で）。
6. **買えないか・採用できないか** — 枯れた OSS との比較を 1 度は行い、自作するならその理由を
   ADR に残す。ただし §3 の依存の警告とセットで判断。
7. **書く** — ここまで来たら、書く量を最小にする（§2）。

## §2 書くと決めた後の最小化

- **YAGNI を機械的に適用する**: 「将来のために」の抽象レイヤ・設定オプション・拡張ポイントは
  書かない。将来必要になった日に書くほうが、要件が確定している分だけ常に安くて正確。
  **その日は大抵来ない**。例外は design doc に明記された差し替えポイントのみ
  （それは設計判断済みの拡張点）。
- **インターフェースは使う側から書く**: 呼び出しコードを先に書き、それが要求する最小の
  シグネチャだけ実装する。実装側から書くと「あるかもしれない用途」のメソッドが増える。
- **削除は commit の主役になれる**: 到達不能コード・無効フラグ・コメントアウトの死体・
  使われないオプションは見つけ次第消す（CLAUDE.md 実装規則「誤った development residue の削除」
  と同一）。履歴は git が持っている。「消したら困るかも」は git への不信であって理由ではない。
- **賢いコードより退屈なコード**: メタプログラミング・過度な DRY・トリッキーなワンライナーは、
  行数を減らして読解コストを増やす本末転倒。最小化するのは**総所有コスト**であって行数ではない。
  3 回未満の重複は、誤った抽象化より安い（間違った共通化は最も高くつく負債）。

## §3 依存追加の判断 — import も 1 行のコードである

ライブラリの採用は「コードを書かない」の変種に見えて、実際は**他人の負債の引き受け**
（EOL・脆弱性・破壊的変更・学習コスト）。判断規準:

- 標準ライブラリ + 数十行で書けるものに依存を足さない（left-pad 事件の教訓）。
- 採用するなら 4 点に答えてから ADR に記録する: 最終リリースが 1 年以内か / メンテナが複数か /
  自分たちが読めるコードか / 撤退経路を描けるか（license・stars・活発度の評価は
  `tech-selection`、調査委譲は pmo-tech-fork）。
- 「試しに入れた」依存が最も危険。試用は branch で完結させ、本流に入れる時点で ADR を書く。

## §4 ハードコードの嗅覚

書いている値・分岐を見て、次のどれかに該当したら手を止める:

- **環境で変わるか** → 接続先・パス・ポートは設定へ。「とりあえず localhost」は必ず本番へ漏れる。
- **秘密か** → 資格情報・鍵はコード・docs・commit のどこにも書かない（安全境界の絶対則。
  検出 gate は最後の網であって作法ではない）。
- **業務が決めた数字か** → 上限・閾値・料率は「誰が・いつ変えたくなるか」を考える。
  変えたい値をコードに書くと、以後すべての変更がリリース案件になる。出所（どの要件・
  どの決定か）をコメントでなく trace（PLAN / design doc）で残す。
- **人間が読む文字列か** → 表示文言の直書きは表記ゆれと i18n 漏れの温床。既存の文言定義に寄せる。
- **特定利用者の匂いがするか** → 特定 tenant / 顧客名での分岐は最悪のハードコード。
  個別対応の分岐は指数的に増殖する。
- **時刻・ロケールの前提か** → 「今日」「JST」「年度は 4 月」の直書きは test-thinking §1-2
  「時が経つ」の攻撃で必ず割れる。注入可能にする。

見つけたが今直せない場合の作法: **黙って残さない**。`debt-register` の規約で「意図した負債」
として起票し、返済条件を書く。記録された負債は経営判断、黙った負債は事故。

## §5 生成 AI 時代の運用規則（HELIX での位置づけ）

- 実装提案の冒頭で、7 段の問いの**どこで止まったかを 1 行宣言**する。「4: 設定で解決可能、
  コード不要」で終わる提案は満点回答である。
- コード量の多さを成果として報告しない。報告すべきは満たした要件 ID（PLAN の generates /
  review_evidence）であって行数ではない。
- adversarial-review の攻撃観点に本原則を使える: 「この実装は 7 段の問いの 3（既存機能）で
  止まれたはずである」は正当な攻撃。設計レビュー時は「足りないもの」だけでなく
  **「書かなくてよかったもの」**を探す。
- 大量生成の誘惑への防波堤: 1 PLAN scope を超える「ついで実装」をしない。ついでに書かれた
  コードは要件 trace がなく（宙吊り）、テストがなく、レビューされない — doctor が赤にする類の
  ものを、そもそも生まない（judgment-core §2 実装中規律と同一）。
