# テスト思考（test-thinking）— 何を疑い、どこを見て、いつ止めるか

テストケースを設計するとき、`docs/test-design/` を書くとき、テスト結果を解釈するとき、
「どこまでテストすれば十分か」を判断するとき、テストが全部通って不安なときに読む。
テストの *what and how*（level / fixture / Vitest 実行）は `testing`、*when*（Red-Green 順序）は
`test-driven-development` が正本。本 pack はその型に入れる**中身の質**、つまり判断フレームを扱う。

## §0 出発点となる態度

**テストの目的は「動くことの確認」ではなく「壊れ方の発見」である。**
確認のつもりで書いたテストは、書き手が想定した壊れ方しか捕まえない。想定内の壊れ方は実装時に
すでに潰されているから、確認テストの多くは最初から通る。最初から通るテストは情報量が少ない。

- テストを書く前に「この機能はどう壊れるか」を最低 3 通り言語化する。言語化できないなら
  対象を理解していない（design doc へ戻る）。
- **全テストが一発で緑になったら喜ばずに疑う。** それは「実装が完璧」よりも「テストが弱い」
  確率のほうがずっと高い。故意にバグを 1 つ入れて赤になるか確かめる（mutation の手動版）。
  赤にならないテストは削除候補。
- 「テストが通った」を「正しい」と報告しない。正確な表現は「宣言したテスト集合に対して
  壊れ方が観測されなかった」である（judgment-core §1-1 `coding ≠ substance` と同じ論理）。

## §1 壊れ方を探す視点カタログ（6 軸）

機械的に列挙するのではなく、対象を見たときに頭の中で走らせる問いとして使う。

1. **入力 — 「ゼロ・一・多・境界・異物」**: 空文字 / 空配列 / null / 0 件。1 件だけのときの
   複数形処理・集計。上限ちょうど / 上限+1 / 実運用の 10 倍。仕様に数字があれば必ず
   「その値・その±1」を突く。**数字が無ければそれ自体が指摘**（判定不能 = adversarial-review の
   攻撃 1 号）。絵文字・サロゲートペア・制御文字・全角数字・先頭末尾の空白・改行入り。
2. **時間と順序 — バグは「間」に棲む**: 操作の順序を入れ替える。途中でやめる（wizard 中断、
   接続断）。二度やる（二重送信・べき等性・retry）。同時にやる（同一リソースの並行編集）。
   時が経つ（失効後の操作、日付跨ぎ・月末・うるう年・timezone）。
3. **状態 — 遷移表の「空欄」を突く**: 状態遷移図があるなら、書かれた遷移ではなく
   **書かれていない組（状態×イベントの空欄）**を叩く。「起きないはず」は実装されていないことが多い。
4. **権限と境界 — 「自分のもの以外」を常に試す**: 他人 / 他テナント / 削除済みのリソース ID を
   直接指定（IDOR）。ロールを 1 段下げて同じ操作。機能テスト中も常時この目を持つ
   （深掘りは `security-and-hardening` / `threat-model`）。
5. **失敗系 — 依存先は必ず落ちる**: 外部 API・DB・filesystem が「遅い / 落ちる / 変な値を返す /
   途中まで成功する」の 4 パターンを想定する。特に「途中まで成功」（3 件中 2 件書いて失敗）は
   補償処理の穴が最も出る。エラーメッセージ自体もテスト対象（内部情報の漏れ、利用者が次の
   行動を判断できるか）。
6. **オラクル — 「正しい」を誰が知っているか**: 期待値が自明でないテストは、期待値の**出所**を
   先に確定する。実装から期待値を逆算したテストは、実装のバグごと固定化する同語反復。
   出所は spec の数式・独立した手計算・別実装・過去の確定データのいずれか。出所が無いなら
   「テストできない要件」であり要件側の欠陥として surface する。

## §2 深さの配分 — リスクベースの実際

全部を同じ深さで診断するのは、浅く広い確認テストの山を作るだけ。深く掘る優先順:

1. **不可逆なもの**: destructive data operation・履歴書換え・外部送信・課金。バグが
   「直せば済む」でないもの（judgment-core §1-3 blast radius と同軸）。
2. **境界にあるもの**: 権限境界・runtime 境界（hook / guard / cross-runtime handover）・外部連携面。
3. **歴史が汚いもの**: 修正が繰り返されているモジュール、`helix doctor` や CI が過去に赤を
   出した箇所。**バグは群生する** — 1 匹見つけた場所の周りをさらに掘るのが統計的に正しい。
4. **新しく複雑なもの**: 分岐の多いロジック・並行処理・状態機械。

浅くてよい対象: 枯れた CRUD、フレームワークが保証する部分、失っても復元容易な表示系。
ここを深掘りするのは、深掘りすべき対象から時間を奪う分だけ有害。

## §3 探索的テストの規律

script 化されたテストの外側を歩くときの規律。「なんとなく触る」ではない。

1. **チャーターを 1 行書いてから始める**: 「handover 境界まわりで stale 状態による誤継続を探す・30 分」。
   目的と時間箱を決めない探索は散歩。
2. **違和感をログせよ、判断は後で**: 再現できなくても記録する（`.helix/audit/` かレポートに残す）。
   再現しない違和感の一定割合は、後で本物のバグの兆候だったと分かる。
3. **1 つ見つけたら変奏する**: 見つけたバグの「兄弟」を探す（同じパターンの別 module、同時期の
   変更）。バグ 1 件の報告よりバグの族の特定のほうが価値が 10 倍ある。
4. **記録は再現手順ではなく再現条件で書く**: 「〜を実行」の羅列ではなく「未 commit の変更がある
   状態で、session 失効後に handover を読むと」。条件で書けないなら、まだ原因の層に届いていない。

## §4 checklist を運用するときの振る舞い

checklist は思考の代替ではなく思考の呼び水である。形骸化との戦い方:

- 各行を「済」にする前に自問する: **「この行を突破する反例を、今の私は 1 つ構成できるか」**。
  構成できてしまうなら済ではない。構成を試みて失敗した記録が「済」の実質的な中身。
- 「済」の根拠は**観測**で書く（「role X で実行し exit 1 を実測」）。「実装されているはず」
  「コードを読んだ限り」は根拠として弱い — 読んだコードと動くコードは別物。
- checklist に無い違和感を見つけたら、行を**追加してから**診断する。リストを育てることが
  診断者の仕事。全行が毎回「済」で流れる期間が続いたら、リストが現実と噛み合わなくなったサイン。

## §5 いつ止めるか — 十分性の判断

テストは論理的には終わらない（欠陥ゼロは証明不能）。止める判断は工学ではなく経済とリスクの
判断であり、**明示的に**行う。

- **止めてよい兆候**: 新しいテストが新しい情報を生まなくなった（直近 N ケースの発見ゼロ）、
  発見が全て既知の族の変奏、残る未探索領域が §2 の「浅くてよい対象」のみ。
- **止めてはいけない兆候**: 発見率がまだ落ちていない、直前に大きな発見があった（バグは群生する）、
  不可逆領域に未探索が残っている。
- 止める判断は**残余リスクの言語化とセットで**行う: 「〜は未検証。壊れた場合の影響は〜、検出は
  〜で可能」。この一文が書けないなら止める判断が甘い。書けたら PLAN の `review_evidence` scope
  または `.helix/audit/` に記録する。

## §6 アンチパターン早見

| アンチパターン | なぜダメか | 代わりに |
|---|---|---|
| happy path 網羅で満足 | 想定内は実装時に潰れている | §1 の視点で壊れ方から設計 |
| 実装を見てから期待値を書く | 同語反復。バグを固定化 | 期待値の独立した出所を先に確定（§1-6） |
| coverage % を目標にする | 行を通る ≠ 検証する。assert なき実行も % は上がる | coverage は「見ていない場所の発見」にのみ使う |
| 落ちたテストを retry で緑にする | 不安定さ自体が並行性バグの兆候 | flaky の原因を 1 件ずつ潰す。無理なら隔離して PLAN 起票 |
| 重大度をつけずに全部報告 | 受け手が判断できず全部無視される | severity-first（judgment-core §4）で仕分けてから報告 |
| 全緑を成果として即報告 | テストが弱い可能性を排除していない | 故意バグ 1 つで赤を確認してから報告 |
