# HELIX 企画書 — 超個人開発システム

> 版: v0.2 ／ status: confirmed（G0.5 PASS 2026-06-28）／ 起票 PLAN: PLAN-L0-01-helix-charter ／ drive: agent
> 改訂: v0.2 (2026-06-28) — Scrum を「大規模開発を機能ユニット分割し L4–L7 を反復着地させる機構」として対象内に訂正（v0.1 で誤って非目標化）。version-up 定義を追加（L1 back-propagation）。
> 位置づけ: 本書は HELIX の L0 企画書（charter）。具体機構（アルゴリズム・スキーマ・gate 配線）は持たず、背景・目的・スコープを高レベルで定め、L1 要求へ trace させる。土台は HELIX Agent Harness（V モデル工程・gate・state DB 実装済み）。

## §1 背景

個人開発を AI に委譲する。旧 HELIX（V モデル起点の品質管理システム）を HELIX ハーネス（TS/Bun、V モデル工程・gate・state DB を実装済み）の上に積み直し、「**人が上流を決めれば、あとは AI がガードレール内で完成まで自走する**」基盤にする。

現ハーネスは「AI を安全に使う検証基盤」としては動くが、**オーケストレーションが弱く**、AI の無人自走には不足している。その克服が HELIX の主眼。完全自動で安全に走らせるには**強い検証**が前提であり、前進には**外部の力を取り込む**が、それは**厳格なセキュリティ**と一体で成立する。

## §2 目的（達成したい状態）

開発者は **企画・要求・デザインモックを直接作り、AI が起草した要件定義を承認する**だけでよい。**要件凍結後は完全自動** — AI が設計 → 実装 → 検証 → PR → クロスレビュー → CI →（失敗なら自動改善）→ マージ → タグ付けまでを、人の逐次承認なしで回す。停止は機械が定める境界（gate 赤／不可逆・破壊的操作）でのみ。

さらにハーネスは**ハーネスメモリを根幹に自己保守・自動アップデート**し、AI は開発だけでなく基盤の維持改善も自走する。

## §3 自律境界（V モデル上・3層）

| 工程 | 担当 |
|---|---|
| **L0 企画 / L1 要求 / L2 デザインモック** | **人が直接作成・指示**（モック(L2)が指示の中心・最後の直接関与） |
| **L3 要件定義** | **AI が起草、人は承認のみ**（人は書かない） |
| **L4 基本設計 → L5 → L6 → L7 実装 → L8 → L9 → L10 → L11 → L12 → L13/L14 ＋ GitHub PR/CI/マージ/タグ** | **AI が完全自動**（人手介入なし。ただし不可逆操作のみ人へ escalate → §4 P8） |

人の創作・指示の中心は**モックまで**。要件定義から先は AI が主導し、人は L3 で承認するだけ。

## §4 スコープ（10 本柱 P0–P9）

> 各柱は高レベルの方向性。機構の詳細は L1 以降で定義する。

- **P0 逸脱受け止めと Forward 収束** — 直線 Forward だけでなく、AI 自走中の逸脱・障害・暴走を受け止め、**必ず Forward へ戻して正本に収束させる**。駆動 workflow（Reverse／Discovery／Incident／Recovery／Refactor／Retrofit／Research／Add-feature）＋ forward_return 戻し規律＋ AI 暴走ガード（lock／budget time-cap／Recovery）。*無人自走の安全弁*。
- **P1 要件承認後フル自動＋連続自律走行エンジン** — §3 の自律境界③を、走り続ける engine（heartbeat／job-queue／budget time-cap／fresh-session 再起動）で完走する。大規模開発は **Scrum**（基本設計 L4 以降、設計を機能ユニット単位に分割し、各ユニットを Forward へ下ろして L4–L7 を反復横断しながら段階着地させる分割実行機構）でスケールさせる。**version-up 定義**（将来版への deferred-but-committed 繰り越し＝`version_target`、リリース／タグ lifecycle。P6 と接続）で版を管理し、今版に入れない作業を失わない。
- **P2 オーケストレーション根本強化＋ループエンジニアリング** — 現ハーネスの弱いオーケストレーションを HELIX の根本強化点とする。サブエージェントを「**プロンプトを複数視点で解釈・検証 → 計画 → 実行 → 検証 → オーケストレーターへ返す**」ループ単位で動かし、orchestrator が統括・統合する。effort／budget 制御を組み込む。設計手法として W モデル（AI エージェントシステムは V を2回）＋型付き内部契約（agent↔tool の request/response 機械検証）を採用検討。
- **P3 強い検証基盤** — HELIX の第一級の強み。完全自動を許容できる安全性の要。**pair_closure（design⇔test を対で凍結、coverage100% 単独 pass 禁止）／片肺禁止／機械 vs AI 判定境界**。成果を外部の真実に照合する。
- **P4 自動保守システム** — drift・劣化・不整合を自動検出し自動修復（自走保守）。**検出 → 自動ルーティング循環**（detection-routing）と**学習ループ**（成功 recipe 蓄積／頻出トラブル → 予防ルール=gate/detector へ昇格）。根幹は P7 ハーネスメモリ。
- **P5 コンテキスト効率向上** — 動的注入・圧縮で「必要分だけ」を渡し、長時間の無人自走を支える。**閾値到達前に handover 要約 → fresh session 移行**、注入予算。
- **P6 GitHub 運用自動化** — commit/push の**ゲート化**（全 gate PASS で authorized push、raw push は fail-close で deny）／PR クロスレビュー自動ワークフロー／**CI 失敗時に自動改善して上げ直し**／タグによるバージョン管理。CI はゲート証跡の検証に専念。
- **P7 ハーネスネイティブ・エージェントメモリ（2層）** — Claude Code 内蔵のメモリ機能は使わず**設計の参考のみ**。ハーネスで動くエージェント自身のメモリを 2 層に分離する：
  - **ハーネスメモリ** — ハーネス自身の改善に使う知見。**保守・アップデートの根幹**（P4 の土台）。
  - **プロジェクトメモリ** — 開発対象プロジェクト固有の知見。
  - 用語の発散を防ぐため **DDD ユビキタス言語（Glossary を SSoT 化）／ Bounded Context** を併せて持つ。
- **P8 外部連携・外部検索（セキュリティ厳格）** — AI は内向きに閉じず、外部（Web／公式 docs／OSS／ツール）を**検索・参照して知見を取り込むことを原則**とする（思い込み・幻覚を外部照合で抑止 → P3 と相乗）。**外部検索で有益な知見はスキル化して自己取り込み**（自己拡張ループ → P4／P7 と接続）。外部連携はすべて**厳重なセキュリティ統制下**（secret 漏洩防止・信頼境界・サンドボックス）。**不可逆操作の escalation 境界**＝本番影響／認証認可／決済／PII／secret／license／schema migration／破壊的データ操作／外部 API・infra 変更のみ人間に戻す。それ以外は自走。
- **P9 HELIX DB 収束（trace／drift／coverage／contract）** — 成果物（doc／code／test／PR）を台帳に収束し、整合を機械追跡する。**DB に収束しない成果は trace／drift／coverage の対象外＝未完了**。AI の「実装したつもり」詐称を防ぎ、影響範囲分析を可能にする資産保全 backbone。横断原則「同一記憶の共有」の機械的実体。

## §5 横断原則：エージェント一貫性

- **全エージェントはハーネスの単一ルールに準拠する**（エージェントごとに独自ルールへ逸脱しない）。
- **全エージェントは同一の記憶を共有する**（P7 の 2 層メモリを共有し、エージェント個別の記憶サイロを作らない）。
- これにより P2 オーケストレーションと P7 メモリ／P9 DB が一貫する。ハーネス既存の `rule-drift`（Codex↔Claude アダプタ乖離の機械検査）を全エージェントへ一般化する位置づけ（具体機構は L1 以降）。

## §6 非目標（スコープ境界）

- **チーム運用・複数人協調は対象外**（`helix team` 相当の人間ロール分担・velocity／sprint ceremony は持たない。ただしサブエージェント分業＝model 割当は P2 に活用する）。**注**: ここでの「Scrum 非採用」はチームの velocity/ceremony に限る。HELIX の Scrum は大規模開発の機能ユニット分割・L4–L7 反復着地機構（P1）であり**対象内**。
- **要件定義（L3）以降を人が直接書くこと、および L4 以降への人手介入は対象外**（人は L3 承認まで）。
- **要件定義より上流の自動化は対象外**（企画・要求・デザインモックは人が確定する）。
- 既存ハーネス（HELIX）の確定済み L1–L7 資産の作り直しはしない（壊さず上に積む）。
- Claude Code 内蔵メモリへの依存（ハーネス外のメモリ機構に開発知見を委ねること）は対象外。
- 具体的なエージェント契約・スキーマ・gate 配線・アルゴリズムの詳細は **L1 以降**で定義（L0 では持たない）。

## §7 成功の方向性（定量 KPI は L0 では強制しない）

代表シナリオ「**人が要件・モックを凍結 → 以降ノータッチ → AI が設計〜実装〜検証〜PR〜クロスレビュー〜CI〜（失敗なら自動改善）〜マージ〜タグまで完走**」が、人手介入最小（理想は要件凍結後ゼロ）で、かつ逸脱・暴走時も P0 で安全に収束しながら回ること。

## §8 L1 への接続（trace seed）

P0–P9 を L1 の業務要求（BR-*）／非機能要求（NFR-*）へ降ろす：逸脱収束・自律境界の正確な定義・オーケストレーション品質・検証網羅（pair_closure）・保守自動度・context 予算・GitHub 自動化・メモリ2層・外部連携セキュリティ・DB 収束の各観点。

## §9 設計上の留意（L0 では決めず L1 で扱う）

- 本件は「AI に開発を委譲するシステム＝AI エージェントシステム」を作る案件のため、ハーネスの **W モデル**（一般システム V ＋ エージェント V の二段、L10 合流）適用が論点。L1 で判定。
- 旧 HELIX（参考: RetryYN/ai-dev-kit-vscode の v3 charter ほか）は本書とほぼ同一ビジョンを持つ設計途中資産。**設計ソースとして参照**するが、実装は HELIX ハーネス（TS/Bun）に積む。
